# Safe Refactor with Property-Based Testing

## Purpose

Safely rewrite or refactor a module by using property-based testing (fast-check) to prove the new code behaves identically to the old code for all possible inputs. The idea: write a bridge that calls both versions, generate thousands of random inputs, and assert the outputs match.

## When to Use

Use this workflow when:
- Rewriting a module or function for performance, readability, or new architecture
- Migrating from one library/framework to another (e.g., switching ORMs)
- Refactoring complex business logic where manual test cases can't cover all edge cases
- Onboarding to a brownfield project and need confidence before changing things

Do NOT use this workflow when:
- The behavior itself needs to change (this proves equivalence, not correctness)
- The function has side effects that can't be isolated (e.g., sends emails, writes to DB)
- The function is trivial (a simple rename or one-liner doesn't need this)

## Prerequisites

Before running this workflow, ensure:
- [ ] fast-check is installed (`pnpm add -D fast-check` or `npm install -D fast-check`)
- [ ] The original code is working and considered "correct" (it's the source of truth)
- [ ] You can identify the public interface of the module being refactored
- [ ] Side effects can be mocked or isolated

## Concepts

**Property-based testing** flips the script on traditional testing. Instead of writing specific test cases ("given input X, expect output Y"), you define *properties* that should always be true ("for ANY valid input, old and new code produce the same output"). fast-check generates hundreds of random inputs to try to break that property.

**The bridge pattern:**

```
Random Input (generated by fast-check)
       │
       ├──▶ Old Code ──▶ Result A
       │
       └──▶ New Code ──▶ Result B
       
Assert: Result A === Result B (for every input)
```

## Steps

### Step 1: Identify the Refactor Boundary

Determine exactly what you're refactoring. Define the boundary:

| Question | Answer |
| :--- | :--- |
| What function(s) are being rewritten? | List them |
| What are the input types? | Define precisely |
| What are the output types? | Define precisely |
| Are there side effects? | List them, plan to mock |
| What's the public API? | The contract that must not change |

**Key rule:** The refactor boundary should be as small as possible. Refactor one module at a time, not the whole system.

### Step 2: Write Input Generators (Arbitraries)

Create fast-check arbitraries that generate valid inputs for the function. This is the most important step — good generators find more bugs.

```typescript
import fc from 'fast-check';

// Example: generating valid task inputs
const taskInputArb = fc.record({
  title: fc.string({ minLength: 1, maxLength: 200 }),
  description: fc.option(fc.string({ maxLength: 5000 })),
  priority: fc.constantFrom('low', 'medium', 'high', 'urgent'),
  dueDate: fc.option(fc.date({ min: new Date('2020-01-01'), max: new Date('2030-12-31') })),
});
```

**Tips for good generators:**
- Use the spec's "Core Types" and "Data Validation" sections to define constraints
- Include edge cases: empty strings, very long strings, boundary dates, zero, negative numbers
- Use `fc.oneof()` to mix valid and edge-case inputs
- For complex objects, build generators from smaller ones using `fc.record()` and `fc.tuple()`

### Step 3: Write the Bridge Test

Create a test file that calls both the old and new implementations:

```typescript
import fc from 'fast-check';
import { processTask as oldProcessTask } from './old/process-task';
import { processTask as newProcessTask } from './new/process-task';

describe('processTask refactor equivalence', () => {
  it('should produce identical output for all valid inputs', () => {
    fc.assert(
      fc.property(taskInputArb, (input) => {
        const oldResult = oldProcessTask(input);
        const newResult = newProcessTask(input);
        
        // Deep equality check
        expect(newResult).toEqual(oldResult);
      }),
      { 
        numRuns: 1000,        // Run 1000 random inputs
        verbose: true,        // Show failing input on failure
        seed: 42,             // Reproducible runs
      }
    );
  });

  it('should handle error cases identically', () => {
    fc.assert(
      fc.property(invalidTaskInputArb, (input) => {
        let oldError: Error | null = null;
        let newError: Error | null = null;

        try { oldProcessTask(input); } catch (e) { oldError = e as Error; }
        try { newProcessTask(input); } catch (e) { newError = e as Error; }

        // Both should throw, or both should succeed
        if (oldError) {
          expect(newError).toBeTruthy();
          expect(newError!.message).toEqual(oldError.message);
        } else {
          expect(newError).toBeNull();
        }
      }),
      { numRuns: 500 }
    );
  });
});
```

### Step 4: Run and Iterate

Run the property tests:

```bash
pnpm test -- --grep "equivalence"
```

**If tests pass:** fast-check couldn't find any input where old and new differ. High confidence the refactor is safe.

**If tests fail:** fast-check found a counterexample — a specific input where old and new produce different results. fast-check will automatically shrink the input to the simplest failing case.

```
Property failed after 47 tests
Counterexample: { title: "", priority: "urgent", dueDate: null }
Shrunk 3 times
```

Fix the new implementation to handle this case, then re-run. Repeat until all properties pass.

### Step 5: Add Targeted Properties

Beyond equivalence, add properties that describe invariants of the system:

```typescript
// Example: output should always have an id
it('should always return a task with an id', () => {
  fc.assert(
    fc.property(taskInputArb, (input) => {
      const result = newProcessTask(input);
      expect(result.id).toBeDefined();
      expect(typeof result.id).toBe('string');
    })
  );
});

// Example: status should always be valid
it('should always return a valid status', () => {
  fc.assert(
    fc.property(taskInputArb, (input) => {
      const result = newProcessTask(input);
      expect(['todo', 'in_progress', 'done']).toContain(result.status);
    })
  );
});
```

### Step 6: UI Regression Check (Web Projects Only)

> Skip this step if the refactor doesn't affect web UI code.

If the refactored module is used by the web UI, run Bombadil before and after to check for UI-level regressions:

**Before refactoring** (baseline):
```bash
bombadil test http://localhost:3000 tests/ui-properties/ --headless --output-path=./bombadil-traces/before/
```

**After refactoring** (comparison):
```bash
bombadil test http://localhost:3000 tests/ui-properties/ --headless --output-path=./bombadil-traces/after/
```

Compare results:
- **No new violations:** The refactor is UI-safe.
- **New violations appeared:** The refactor broke something in the UI. The violation report tells you which property failed and what action triggered it. Fix before proceeding.

Even without custom Bombadil specs, the default properties (`no_uncaught_exceptions`, `no_console_errors`, etc.) catch UI regressions caused by refactored logic throwing errors, returning unexpected values, or breaking rendering.

### Step 7: Replace and Clean Up

Once all property tests pass:

1. Replace the old module with the new one
2. Keep the property tests — they become regression tests
3. Remove the bridge (old import) but keep the property assertions
4. Run the full test suite to ensure nothing else broke
5. Update the spec if the refactor revealed any spec inaccuracies

## Property Ideas by System Type

| System Type | Properties to Test |
| :--- | :--- |
| **Data transformation** | `transform(input)` always produces valid output shape |
| **Validation** | Valid inputs always pass, known-invalid inputs always fail |
| **Serialization** | `deserialize(serialize(x))` equals `x` (round-trip) |
| **Sorting/Filtering** | Output length <= input length, output is sorted, all items match filter |
| **Math/Calculation** | Results within expected bounds, commutative/associative properties hold |
| **State machines** | All transitions produce valid states, no invalid state reachable |
| **Parsers** | `parse(format(x))` equals `x`, invalid input always throws |
| **API handlers** | Response shape matches spec, status codes correct for input class |

## fast-check Cheat Sheet

```typescript
// Primitives
fc.integer()                          // any integer
fc.integer({ min: 0, max: 100 })      // bounded integer
fc.string()                           // any string
fc.string({ minLength: 1 })           // non-empty string
fc.boolean()                          // true or false
fc.double()                           // any float
fc.date()                             // any Date

// Combinators
fc.constantFrom('a', 'b', 'c')       // one of these values
fc.option(fc.string())                // string | null
fc.array(fc.integer())                // number[]
fc.array(fc.integer(), { minLength: 1, maxLength: 10 })

// Objects
fc.record({                           // typed object
  name: fc.string(),
  age: fc.integer({ min: 0, max: 150 }),
})

// Advanced
fc.oneof(fc.string(), fc.integer())   // union type
fc.tuple(fc.string(), fc.integer())   // [string, number]
fc.uniqueArray(fc.integer())          // array with no duplicates

// Configuration
fc.assert(fc.property(arb, (val) => { ... }), {
  numRuns: 1000,      // number of random inputs
  verbose: true,      // show counterexample details
  seed: 42,           // reproducible
  endOnFailure: true, // stop at first failure
});
```

## Success Criteria

The workflow is complete when:
- [ ] Property tests pass with 1000+ runs
- [ ] Equivalence bridge confirms old and new produce identical output
- [ ] Targeted invariant properties all pass
- [ ] (If web UI) Bombadil default + custom properties pass with no new violations vs. baseline
- [ ] Old module replaced with new module
- [ ] Full test suite passes
- [ ] Property tests kept as regression tests

## Troubleshooting

### fast-check finds too many trivial failures

**Symptom:** Tests fail on inputs that are technically invalid (empty strings, etc.)

**Solution:** Tighten your arbitraries to only generate valid inputs. Use `fc.pre()` to add preconditions, or constrain generators with min/max.

### Tests pass but you're not confident

**Symptom:** 1000 runs pass but the function is complex

**Solution:** Increase `numRuns` to 10000 or 100000. Add more targeted properties. Review the generator — is it actually producing diverse inputs? Use `fc.statistics()` to see the distribution.

### Side effects make bridging impossible

**Symptom:** Function writes to database, sends HTTP requests, etc.

**Solution:** Extract the pure logic from the side-effectful wrapper. Test the pure logic with property tests. Test the side-effectful wrapper with traditional integration tests.

## Notes

- Property-based testing doesn't replace unit tests — it complements them
- The best property tests come from reading the spec's "Core Types" and "Data Validation" sections
- Keep the bridge tests even after refactoring — they catch future regressions
- fast-check's shrinking is magic: when it finds a failure, it automatically finds the simplest input that reproduces it
- For async functions, use `fc.assert(fc.asyncProperty(...))` instead
